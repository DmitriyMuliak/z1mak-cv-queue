# Так як ми перевіряємо зараз ліміти у api то ми навіть не візьмемо задачу якщо rpm моделі перевищені тому нам не страшно що ми перевантажимо зовнішній api (rpm).

Але питання - як ми можемо перевіряти -
1- user rpm
2- user rdp
3- user concurrency
4- model rdp

Але model rpm не буде відсікати запит а просто буде класти задачу в чергу та наш bull mq буде після оновлення model rpm буде брати певну кількість задач з пулу та запускати - але як же нам тоді керувати concurrency option коли додаємо задачу та загалом коли bull mq буде вибирати з черги певну кількість задач ?

---

Це дуже глибокі і правильні питання, що демонструють розуміння конфлікту між **жорсткими лімітами (Rate Limiting)** та **гнучким управлінням навантаженням (Queue Management)**.

Ваша ідея про те, що **Model RPM** має бути керовано чергою, а не API, є **архітектурно кращою** для забезпечення високої доступності та уникнення 429-х.

Ось як ми можемо розділити відповідальність та відповісти на ваші питання.

---

## 1\. 🚦 Як перевіряються поточні ліміти (API)

Ви абсолютно праві: оскільки ми перевіряємо більшість лімітів у **`jobs.ts` (API)**, ми гарантуємо, що **більшість перевантажень буде відсічено до черги**.

### Ліміти, які залишаються в API:

1.  **✅ User RPM (Ключ `KEYS[3]`):** **Потрібно залишити.** Захищає від спаму користувача. Якщо користувач зробив 1000 запитів за хвилину, ми його блокуємо, незалежно від стану моделі.
2.  **✅ User RPD (Ключ `KEYS[4]`):** **Потрібно залишити.** Це **квота** користувача. Якщо квота вичерпана, API має відмовити.
3.  **✅ User Concurrency (Ключ `KEYS[5]`):** **Потрібно залишити.** Захищає Worker-и від монополізації одним користувачем.
4.  **✅ Model RPD (Ключ `KEYS[2]`):** **Потрібно залишити.** Це **глобальна квота** для моделі. Якщо квота (наприклад, 100000/день) вичерпана, ми припиняємо приймати завдання.

### Ліміт, який потрібно **видалити** з API:

- **❌ Model RPM (Ключ `KEYS[1]`):** **Має бути видалений.** Саме цей ліміт повинен керуватися чергою.

---

## 2\. ⏳ Як керувати Model RPM через BullMQ

Щоб перетворити Model RPM з жорсткого ліміту на механізм **контролю швидкості споживання черги**, ми можемо використати BullMQ **Rate Limiter** або **Priority Queue**.

### 2.1. Використання BullMQ Rate Limiter (Рекомендовано)

Ми видаляємо перевірку `Model RPM` з Lua-скрипту і переносимо її на рівень черги:

**1. Зміна конфігурації черги (`queue.ts`):**

```typescript
// queue.ts (Приклад налаштування Rate Limiter для черги)
export const queue = new Queue('ai-jobs', {
  connection: redis,
  defaultJobOptions: {
    // Встановлюємо загальне обмеження швидкості для ВСІХ Worker-ів,
    // які працюють з цією чергою.
    // Наприклад: 100 завдань на 60 секунд.
    rateLimit: {
      max: 100, // Максимальна кількість завдань
      duration: 60000, // за цей інтервал (60 секунд)
      // bounceBack: true, // Якщо ліміт перевищено, завдання залишається в черзі
    },
    // ... інші опції
  },
});
```

- **Наслідок:** Якщо Worker-и спробують обробити завдання швидше, ніж 100/хв, BullMQ **автоматично призупинить** їх і не видаватиме їм нові завдання, доки ліміт не скинеться.
- **Перевага:** Черга виступає в ролі **буфера**, а не відсікача. Завдання накопичуються, але виконуються з контрольованою швидкістю.

### 2.2. Як це впливає на Model RPM

1.  **API (`jobs.ts`):** Тепер він завжди приймає завдання (якщо інші ліміти OK) і кладе їх у чергу.
2.  **BullMQ:** Обмеження `rateLimit: 100/60000` гарантує, що зовнішній API ніколи не отримає більше 100 запитів на хвилину, незалежно від того, скільки Worker-ів ви запустили. Це є **ідеальним механізмом** для керування Model RPM.

---

## 3\. 🎯 Керування Concurrency Option

Ви правильно помітили, що тепер є два рівні Concurrency:

### А. **User Concurrency (API/Lua):**

- **Керування:** `userActiveJobs` (ZSET).
- **Призначення:** Запобігає користувачу мати понад 5 (або N) **активних** завдань.
- **Механізм:** **Жорстке відхилення** (429) в API.

### Б. **Worker Concurrency (BullMQ/Worker):**

- **Керування:** Опція `concurrency` у `worker.ts`.
- **Призначення:** Запобігає **одному Worker-процесу** відкривати надто багато з'єднань одночасно.
- **Механізм:** **М'яке обмеження** (Flow Control) всередині Worker-а.

**Важливий висновок:**

- **`concurrency` у Worker-і** (`worker.ts`) визначає, скільки HTTP-запитів може робити **кожен процес**. Це потрібно для захисту від 429-х на рівні процесу (як обговорювалося раніше).
- **`rateLimit` у Queue** (`queue.ts`) визначає, скільки завдань буде оброблятися **всіма процесами разом** (Model RPM).

Обидва параметри є необхідними для стабільності, але вони вирішують різні проблеми\!

# ⛩️ Опис НОВОЇ архітектури лімітів

У вашій архітектурі ми маємо **п'ять** основних лімітів, які працюють на трьох рівнях (API, Queue, Worker):

---

## 🚦 Ліміти, які ви будете використовуєте (API-рівень)

Ці ліміти відсікають трафік **до** того, як завдання потрапляє в чергу. Вони керуються Lua-скриптом у `jobs.ts`.

| Параметр             | Повна Назва          | Відповідальність                                                                                                                                     |
| :------------------- | :------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------- |
| **User RPD**         | Rate Per Day (Квота) | **Захист білінгу/квоти користувача.** Гарантує, що користувач не перевищить денну квоту (наприклад, 10 завдань на день).                             |
| **Model RPD**        | Rate Per Day (Квота) | **Захист загального бюджету моделі.** Гарантує, що вся система не перевищить денну квоту, надану зовнішнім провайдером.                              |
| **User Concurrency** | Одночасність         | **Захист пулу Worker-ів від монополії.** Обмежує, скільки завдань **один користувач** може мати у статусі "активне виконання" + "в черзі" одночасно. |

---

## 🚀 Ліміти, які ви ВСТАНОВЛЮЄТЕ для керування швидкістю (Queue/Worker-рівень)

Ці параметри відповідають за **контроль потоку (Flow Control)** та **швидкість обробки** завдань, які вже потрапили в чергу.

### 1\. Limiter (BullMQ Rate Limiter)

Це механізм, який контролює **загальну швидкість витягування завдань з черги всіма Worker-ами разом**.

| Параметр       | Призначення             | Пояснення                                                                                                        |
| :------------- | :---------------------- | :--------------------------------------------------------------------------------------------------------------- |
| **`max`**      | **Кількість операцій.** | Визначає **максимальну кількість** завдань, які можуть бути оброблені **всіма Worker-ами** за період `duration`. |
| **`duration`** | **Часовий інтервал.**   | Визначає **довжину часового вікна** (у мілісекундах), за яке рахується ліміт `max`.                              |

**Головна відповідальність:** **Model RPM (Rate Per Minute).** Встановлюється трохи нижче офіційного RPM ліміту Gemini, щоб BullMQ виступав як **буфер безпеки** і запобігав 429-м помилкам. Якщо Worker-и намагаються працювати швидше, BullMQ призупиняє видачу завдань, поки вікно `duration` не скинеться.

### 2\. Concurrency (Worker Concurrency)

Цей параметр встановлюється у файлі `worker.ts` при ініціалізації:

```typescript
new Worker(queueName, processor, { concurrency: N, ... })
```

| Параметр          | Призначення                | Пояснення                                                                                        |
| :---------------- | :------------------------- | :----------------------------------------------------------------------------------------------- |
| **`concurrency`** | **Локальна одночасність.** | Визначає **максимальну кількість** завдань, які може обробляти **один процес Worker** одночасно. |

**Головна відповідальність:** **Захист окремого Worker-процесу.** Це запобігає перевантаженню:

1.  **Зовнішнього API:** Зменшує ймовірність **сплесків (bursts)** HTTP-запитів, які можуть спричинити тимчасові 429-ті.
2.  **Внутрішнього Worker:** Запобігає вичерпанню локальних ресурсів (сокети, пам'ять) в одному процесі.

### 3\. ⏱️ Backoff/Retry Logic (Додатковий, але важливий)

Хоча це не ліміт, це критичний параметр, який ви встановлюєте у `jobs.ts` при додаванні завдання:

| Параметр      | Призначення                | Пояснення                                                                                                                                 |
| :------------ | :------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------- |
| **`backoff`** | Стратегія повторних спроб. | Визначає, як довго BullMQ має чекати, перш ніж знову запустити завдання, яке завершилося помилкою (наприклад, **429 Too Many Requests**). |

**Головна відповідальність:** **Адаптивне відновлення після 429-х.** Якщо ліміти RPM були неточно розраховані або Gemini API сповільнився, Worker просто кидає виняток, і BullMQ чекає `delay` (наприклад, 10 секунд), перш ніж спробувати знову.

# 📈 План покращення

Це дуже важливе уточнення\! Ви абсолютно праві, статичний `rateLimit` або `concurrency` для всієї черги не підходить для системи з різними моделями та різною тривалістю виконання завдань.

Для вирішення цієї проблеми нам потрібно використовувати **три рівні контролю**, щоб забезпечити гнучкість і стабільність:

---

## 1\. 🥇 Багаторівнева Конкурентність (Multiple Queues / Dynamic Concurrency)

Оскільки час виконання завдань (latency) для моделей сильно відрізняється, найкраще рішення — розділити їх на **окремі черги**.

### 1.1. Розділення на Черги за Типом Навантаження

Замість однієї черги `ai-jobs`, створіть дві або більше, засновані на їхній швидкості/витратах:

| Тип Моделі        | Приклад Черги | Переваги                                               |
| :---------------- | :------------ | :----------------------------------------------------- |
| **Швидкі/Lite**   | `lite-jobs`   | Швидкі завдання не застрягають за повільними.          |
| **Повільні/Hard** | `hard-jobs`   | Можна виділити менше Worker-ів, щоб захистити ресурси. |

### 1.2. Визначення Concurrency для Worker-а

Опція `concurrency` у Worker-і (наприклад, `worker-lite.ts`) тепер буде визначатися, виходячи з **очікуваної тривалості** завдання для цієї групи моделей.

- **Формула оцінки:** Оцінюйте, скільки I/O-запитів ви можете безпечно відкрити до зовнішнього API **на один процес Worker**.
  - Якщо модель **швидка** (1-3 сек) і має високий RPM, ви можете безпечно встановити `concurrency: 10-20`.
  - Якщо модель **повільна** (15+ сек), варто обмежити `concurrency: 2-5`, щоб уникнути накопичення сотень незавершених HTTP-з'єднань.

---

## 2\. ⚙️ Контроль Model RPM (Rate Limit)

Ваше ключове спостереження: "ми не можемо статично вказати один і той самий rate limit для всіх моделей."

**Рішення:** Замість загального `rateLimit` на черзі, ми використовуємо **динамічний `rateLimit` на рівні Job Options** при додаванні завдання.

### 2.1. Додавання Rate Limit при `queue.add()`

BullMQ дозволяє передати опцію `rateLimit` безпосередньо в `queue.add()`, використовуючи **`limiter`**.

1.  **Конфігурація Model RPM у Redis:** У `jobs.ts` ви вже отримуєте `modelRpm` (наприклад, 300/хв) з Redis.
2.  **Перенесення в Job Options:** Передаємо це значення при постановці в чергу.

<!-- end list -->

```typescript
// src/routes/jobs.ts
// ... (всередині циклу або після вибору selectedModel)

const modelRpm = Number(modelLimits.rpm ?? 0); // Наприклад, 300
const modelRateLimitDuration = 60000; // 1 хвилина

fastify.queue.add(
  'ai-job',
  {
    /* payload */
  },
  {
    // ... інші опції (jobId, attempts, backoff)
    limiter: {
      max: modelRpm, // 300
      duration: modelRateLimitDuration, // 60000 ms
    },
  }
);
```

### 2.2. Як це працює

BullMQ оброблятиме це як **динамічне обмеження швидкості**. Якщо модель А має ліміт 100/хв, а модель Б — 500/хв, BullMQ гарантує, що завдання для моделі А будуть виконуватися повільніше, незалежно від того, наскільки швидко Worker-и можуть їх обробляти.

---

## 3\. ⚖️ Керування Швидкістю (Масштабування)

**Параметр, який реально керує кінцевою швидкістю** (Throughput) — це кількість **запущених процесів Worker** (`flyctl scale count worker=X`).

- **Якщо завдання повільні:** Запустіть менше Worker-ів, щоб система не накопичувала велику кількість незавершених завдань.
- **Якщо завдання швидкі:** Запустіть більше Worker-ів, щоб "з'їсти" чергу швидше.

Таким чином, ви використовуєте:

1.  **API/Lua** для **квот** (RPD, User Concurrency).
2.  **BullMQ `limiter`** для **швидкості** (Model RPM).
3.  **`concurrency` у Worker-і** для **захисту** зовнішнього API та внутрішніх ресурсів.
4.  **Масштабування** для **фактичної продуктивності**.

# 🚦 Накопичення завдань у черзі

Ви описуєте ідеальний сценарій для використання **BullMQ Rate Limiter (`limiter`)**, який саме так і працює.

Вам **не потрібно** писати жодної додаткової логіки `delay` або `sleep` у Worker-і. BullMQ автоматично керує цим процесом.

Ось як це реалізується, щоб прийняти 115 завдань, але обробляти лише 15 за хвилину, використовуючи механізм **Flow Control** черги.

---

## 🛠️ Реалізація "Накопичуй та Контролюй"

### 1\. 📝 Постановка завдань (API, `jobs.ts`)

API-сервер приймає всі 115 запитів (якщо вони пройшли RPD та User Concurrency перевірки) і додає їх у чергу, але з динамічним лімітом RPM, взятим для цієї моделі (наприклад, 15 RPM).

Припустимо, ви використовуєте окремі черги для моделей, або динамічно передаєте `limiter` опції в `add()`.

```typescript
// src/routes/jobs.ts
const MODEL_RPM_LIMIT = 15; // 15 завдань на хвилину
const RATE_DURATION_MS = 60000; // 1 хвилина

// Постановка завдання в чергу з лімітером:
await fastify.queue.add(
  'ai-job',
  { jobId /* ... */ },
  {
    jobId,
    // ... інші опції (attempts, backoff)
    limiter: {
      max: MODEL_RPM_LIMIT, // 15
      duration: RATE_DURATION_MS, // 60000 мс
    },
  }
);
// API успішно повертає jobID 115 разів.
```

### 2\. 🚀 Обробка завдань (Worker, `worker.ts`)

Ваш Worker (`worker.ts`) не знає і не повинен знати про Rate Limit. Він просто намагається взяти завдання:

```typescript
// src/worker.ts
const worker = new Worker(
  'ai-job',
  async (job) => {
    // BullMQ видасть це завдання лише тоді, коли дозволить лімітер
    console.log(`Обробка завдання ${job.id} розпочато.`);
    await modelProvider.execute(job.data);
    // ...
  },
  {
    // Concurrency може бути високим (наприклад, 20),
    // але це лише максимальна кількість, яку може обробляти ОДИН процес,
    // якщо це дозволено лімітером.
    concurrency: 20,
    connection: redis,
  }
);
```

### 🧠 Як BullMQ гарантує RPM (Магія Flow Control)

BullMQ використовує свій внутрішній механізм **Rate Limiter (на основі Redis)**:

1.  **Запит на завдання:** Кожен Worker постійно просить чергу: "Дай мені наступне завдання".
2.  **Перевірка лімітера:** BullMQ дивиться на поточний лічильник:
    - **Якщо ліміт не вичерпано:** (менше 15 завдань за останні 60 секунд) -\> BullMQ **видає** завдання Worker-у.
    - **Якщо ліміт вичерпано:** (вже оброблено 15 завдань, і 60 секунд ще не минуло) -\> BullMQ **змушує Worker-а чекати (Delay)** і не видає жодних нових завдань, поки вікно часу не скинеться.
3.  **Ефект:** З 115 завдань, BullMQ гарантує, що лише **15 завдань будуть віддані** Worker-ам протягом першої хвилини. Наступні 15 будуть віддані протягом другої хвилини, і так далі. Завдання **накопичуються в черзі**, але **виконуються з контрольованою швидкістю**.

**Висновок:** BullMQ Rate Limiter ідеально реалізує вашу логіку "накопичити 115 завдань і обробляти N за хвилину".

# MINUTE_TTL = 70 sec - why ?

Ви посилаєтеся на використання `MINUTE_TTL = 70` секунд для лімітів, які мають скидатися кожну хвилину (60 секунд).

Це є **стандартною best practice** у розподілених системах, що використовують Redis для Rate Limiting, і використовується для запобігання **Boundary Condition Race** (проблемі на межі часового вікна).

---

## 💡 Причина: Запобігання Boundary Condition Race

Якщо ви встановлюєте TTL (Time To Live) **рівно 60 секунд**, може виникнути проблема:

### ❌ Сценарій з TTL = 60s (Неправильно)

1.  **t = 0.000 с:** Користувач робить запит. Redis встановлює лічильник: `counter = 1`, `TTL = 60s`.
2.  **t = 59.999 с:** Користувач робить останній (наприклад, 10-й) запит.
3.  **t = 60.001 с:** Лічильник **скидається** (TTL минув).
4.  **t = 60.002 с:** Користувач миттєво робить 1-й запит наступної хвилини.

У цьому випадку користувач фактично зробив **20 запитів** (10 + 10) за період **трохи більше 60 секунд**. Це **правильно**, якщо ми використовуємо стратегію **Fixed Window** (яка скидається рівно через 60 секунд).

### 🛡️ Сценарій з TTL = 70s (Sliding Window/Захист від Edge Case)

Проте, використовуючи **70 секунд**, ми створюємо невелике **захисне вікно** (Guard Band) у 10 секунд.

У розподілених системах ми часто використовуємо стратегію **Sliding Window** (Ковзне вікно). У ній TTL має бути довшим за вікно, яке ми контролюємо, щоб гарантувати, що в межах будь-якої 60-секундної вибірки користувач не перевищить ліміт.

**Практичне застосування у вашому коді:**

Ваш Lua-скрипт, який реалізує лімітування, вимагає, щоб TTL був трохи довшим, ніж вікно лімітування. Додавання 10 секунд (70s замість 60s) гарантує, що:

1.  Лічильники не скинуться **занадто рано** через мікроскопічні затримки мережі або неточність годинника між серверами.
2.  Ми даємо додатковий час для **завершення** всіх операцій, що почалися в рамках вікна.

**Резюме:**

Ми використовуємо `MINUTE_TTL = 70` як **технічний буфер (Technical Buffer)**, щоб уникнути помилок через неточність синхронізації часу і гарантувати, що навіть у найгіршому випадку 60-секундний ліміт буде суворо дотриманий.

# Як реалізується, щоб прийняти 115 завдань, але обробляти лише 15 за хвилину, використовуючи механізм Flow Control черги.

## як відсікати нові завдання в API layuer якщо я бачу шо у BullMQ вже накопичилось 115 задач і більше в чергу я не хочу класти поки не стане 115 <

Ви ставите завдання, яке вимагає **зворотного тиску (Backpressure)** від черги до API. Це класичний сценарій, коли потрібно обмежити довжину черги, щоб уникнути надмірного використання пам'яті Redis або занадто довгого часу очікування для користувачів.

Для цього нам потрібно перевірити **поточну кількість завдань у черзі** перед тим, як додавати нове.

---

## 🛠️ Реалізація Backpressure (Обмеження Довжини Черги)

### 1\. 🔍 Метод Перевірки Довжини Черги

BullMQ надає метод `getJobCounts()` або `getWaitingCount()`, але ці операції **не є атомарними** з операцією `add()`, тому потрібно бути обережним.

Для простої перевірки довжини черги, яка є достатньою для Backpressure, ми можемо використати `getWaitingCount()`:

```typescript
// jobs.ts (викликаємо в API layer)

const MAX_QUEUE_LENGTH = 115; // Ваше обмеження

// 1. Отримання кількості очікуючих завдань
const waitingCount = await fastify.queue.getWaitingCount();

if (waitingCount >= MAX_QUEUE_LENGTH) {
  // 2. Якщо черга заповнена, відхиляємо запит
  return reply.status(429).send({
    ok: false,
    error: 'QUEUE_OVERFLOW',
    message: `Черга переповнена. Наразі очікують ${waitingCount} завдань.`,
  });
}

// 3. Якщо все OK, додаємо завдання
await fastify.queue.add(/* ... */);
```

### 2\. ⚠️ Де це розмістити у `jobs.ts`

Цю перевірку потрібно виконати **після** всіх перевірок лімітів користувача (User RPD, User Concurrency), але **до** фактичного додавання завдання в чергу (`fastify.queue.add()`):

```typescript
// src/routes/jobs.ts (Приблизно 150-й рядок)

// ... (1. Lua-скрипт - Перевірка Concurrency, RPD/RPM)
// ... (2. Якщо Lua OK, ми дійшли до цього місця)

// =======================================================
// 3. НОВИЙ ЕТАП: ПЕРЕВІРКА DURATION/QUEUE LIMIT
// =======================================================

const MAX_QUEUE_LENGTH_PER_MODEL = 115;
// Використовуйте окрему чергу (наприклад, fastify.queueLite, fastify.queueHard)
// або одну чергу, але тоді це буде загальний ліміт.

const waitingCount = await fastify.queue.getWaitingCount();

if (waitingCount >= MAX_QUEUE_LENGTH_PER_MODEL) {
  // Якщо черга занадто довга, відхиляємо, щоб час очікування не був занадто великим.
  return reply.status(503).send({
    ok: false,
    error: 'QUEUE_TOO_LONG',
    message: `Сервіс тимчасово недоступний. Очікування: ${waitingCount} завдань.`,
  });
}

// 4. Додавання завдання в чергу з limiter
await fastify.queue.add(/* ... */);

// ...
```

### 3\. ⚖️ Обговорення Атомарності та Конкурентності

- **Атомарність:** На відміну від Redis `INCR` (який завжди атомарний), BullMQ `getWaitingCount()` та `add()` не є єдиною атомарною операцією.
- **Практичний наслідок:** Теоретично, якщо одночасно прийдуть 1000 запитів, вони можуть всі пройти цю перевірку `if (waitingCount >= 115)` і додати завдання, перш ніж `waitingCount` оновить своє значення в Redis.
- **Рішення:** Для Backpressure це зазвичай **прийнятно**. Навіть якщо черга тимчасово зросте до 130-150 завдань, це не є критичною проблемою, оскільки мета — не абсолютна точність, а запобігання нескінченному зростанню черги.

**Висновок:** Використання `getWaitingCount()` в API-шарі є стандартним і простим способом реалізації зворотного тиску, що обмежує максимальний час очікування для нових користувачів.
