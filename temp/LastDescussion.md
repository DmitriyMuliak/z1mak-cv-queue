TTL для userActiveJobs
Зараз 70s — цього не вистачає, якщо ми дозволяємо накопичення в черзі. Мінімальний rpm у сиді — 2 rpm для pro2dot5. Як компроміс проти зомбі-локів ми й так видаляємо слот на завершенні, а довший TTL тільки страхує саме чергове очікування.
Класти modelRpm у payload/opts

Плюси: немає head-of-line blocking між повільними/жорсткими та швидкими/лайтовими моделями; простіше ставити різні rateLimit/concurrency/довжину черги; можна масштабувати воркери адресно.
Мінуси: більше інстансів Queue/QueueEvents, більше конфігів, потрібно правильно роутити job у API та синхронно оновлювати ліміти для кожної черги.

maxQueueLength саме не опція queue.add. Це перевірка, яку робимо в API перед додаванням задачі:

Десь поруч із queue.add у src/routes/jobs.ts беремо ліміт для моделі (мапа в коді / env / Redis).
Піднімаємо лічильник очікування для цієї моделі (queue:waiting:{model} або queue:waiting:{type}) і дивимось, чи не перевищено maxQueueLength.
Якщо перевищено — відкатуємо лічильник і відповідаємо 429/503.
Якщо OK — додаємо job у queue.add.
У воркері при старті (і в finally) — зменшуємо цей лічильник, щоб не зависав.
Тобто maxQueueLength — не параметр add(), а ваша власна перевірка/лічильник в API-шарі до постановки в чергу.
