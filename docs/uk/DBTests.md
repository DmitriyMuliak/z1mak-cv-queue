Тестування баз даних у сучасних фронтенд та фулстек архітектурах (особливо з Supabase/PostgreSQL) ділиться на три рівні. Команда `supabase test db` використовує інструмент **pgTAP**.

Ось як це влаштовано з точки зору архітектури:

### 1. Модульне тестування (Unit Tests via pgTAP)

Це тестування логіки безпосередньо всередині БД: чи існують таблиці, чи правильно працюють RLS (Row Level Security), тригери та збережені функції.

- **Інструмент:** `pgTAP` (написано на SQL).
- **Що перевіряємо:**
- Чи має колонка `email` обмеження `UNIQUE`.
- Чи повертає функція очікуваний результат.
- Чи заборонено анонімному користувачу читати таблицю `profiles`.

**Приклад тесту (`tests/db/hello_world.sql`):**

```sql
begin;
select plan(2); -- плануємо 2 тести

-- Перевірка структури
select has_column('public', 'profiles', 'id', 'Таблиця profiles повинна мати id');

-- Перевірка RLS
select tests.authenticate_as('authenticated_user_id');
select is_empty('select * from private_data', 'Користувач не бачить приватні дані');

select * from finish();
rollback;

```

---

### 2. Інтеграційне тестування (Integration Tests)

Тестування взаємодії вашого коду (Node.js/TS) з базою. Ви запускаєте локальний контейнер Supabase і проганяєте тести через Jest/Vitest.

- **Інструмент:** `Vitest` + `Supabase Client`.
- **Підхід:** 1. Запуск `supabase start`.

2. Виконання тестів, які реально записують дані в локальну БД.
3. Скидання стану БД (Reset) після тестів.

---

### 3. Тестування міграцій (Migration Testing)

Перевірка того, що ваші SQL-міграції застосовуються чисто і не "ламають" існуючі дані.

- **Команда:** `supabase db reset`.
- **Суть:** Supabase намагається застосувати всі файли з папки `supabase/migrations` на чисту БД. Якщо є синтаксична помилка або конфлікт — тест провалено.

### Спірні питання:

1. **Де писати логіку?** Якщо ви пишете багато логіки на рівні БД (тригери, функції), вам **необхідно** використовувати `pgTAP`. Якщо база — це просто сховище, достатньо інтеграційних тестів на рівні API.
2. **Seed дані:** Для тестів важливо мати передбачуваний стан. Використовуйте `supabase/seed.sql` для наповнення бази перед тестами.
